1. Project Summary

Defend Your Thesis is a survival-defense game built with HTML, CSS, JavaScript, and the Canvas 2D API. The player protects a thesis core from bugs, deadlines, and Reviewer #2. It runs at a stable GitHub Pages URL with no build step or server.

3. Gameplay

The player selects a candidate, moves around the arena, fires citations, protects thesis integrity, builds a combo, collects power-ups, and survives progressively harder waves.

4. Architecture

Input events
    |
Update player -> bullets -> enemies -> collisions
    |
Wave manager -> score -> health -> power-ups
    |
Render arena -> thesis -> entities -> effects -> HUD

The game uses requestAnimationFrame and delta-time movement. A 0.033-second cap prevents large jumps after a browser tab returns from the background. Data objects define character and enemy balance separately from runtime logic.

5. AI-Assisted Architecture

An AI coding agent inspected the original static repository and helped convert the assignment brief into modules: character selection, Canvas engine, screen states, multi-input controls, wave management, collision detection, documentation, and tests.

A framework-free implementation was selected because the project has one game scene, no server state, and an existing GitHub Pages deployment. This reduced dependencies and kept the source easy to review.

6. AI-Assisted Problem Solving

Collision detection

Circle-based collision checks are used for bullets, enemies, the player, power-ups, and the thesis: distance(a, b) < radius(a) + radius(b).

Input unification

Keyboard and on-screen direction controls write to the same key state. Mouse clicks and the touch fire button use the same firing state. Core game logic therefore does not depend on one physical device.

Browser audio

Web Audio is initialized only after the Begin Defense button is pressed, satisfying browser autoplay rules. Short synthesized tones avoid copyrighted assets and network downloads.

8. Challenges

Character balance is stored as data so it can be tuned without rewriting game logic. Mobile input required an auto-aim option because touch devices cannot comfortably combine movement and precise pointer aiming. Readability is supported through color, shape, labels, health bars, particles, screen shake, and sound.

9. Limitations

  • Enemies use direct movement instead of pathfinding.
  • The game currently contains one arena.
  • Difficulty needs broader user playtesting.
  • Best scores remain local to one browser.

10. Reflection

The AI agent was most useful for translating requirements into a testable architecture and checking edge cases across game states. Human judgment was still necessary for scope, visual style, honest documentation, and deciding whether each feature improved the product.

The result goes beyond an isolated prompt: it has explicit requirements, persistent state, input handling, collision rules, time-based movement, responsive controls, documentation, and a repeatable test plan.
