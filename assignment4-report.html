1. Project Summary

Defend Your Thesis is a survival-defense game built with HTML, CSS, JavaScript, and the Canvas 2D API. The player protects a thesis core from bugs, deadlines, and Reviewer #2. It runs at a stable GitHub Pages URL with no build step or server.

3. Gameplay

The player selects a candidate, moves around the arena, fires citations, protects thesis integrity, builds a combo, collects power-ups, and survives progressively harder waves. After every cleared wave, a three-choice research draft lets the player shape a different build.

CharacterStrengthTrade-off
AnalystBalanced statisticsNo extreme advantage
HackerFast movement and fireLow damage per shot
WriterHigh citation damageSlow movement and fire

Active abilities

The Analyst freezes every current enemy, the Hacker releases a radial Hotfix Pulse, and the Writer starts a temporary Citation Storm. Every character can also dash with Shift to reposition and briefly avoid boss projectiles.

Research upgrades

The upgrade pool includes damage, fire rate, multi-shot, piercing, critical chance, movement, cooldown reduction, armor, renewable thesis shields, automatic repair, projectile velocity, and orbiting research assistants.

Special enemies

Replicators split on death, Paywalls protect their health with a shield, Reviewer #2 follows a curved approach, and the Committee Chair appears every fifth wave with ranged volleys and summoned threats.

4. Architecture

Input events
    |
Update player -> abilities -> drones -> projectiles
    |
Collisions -> score -> shield -> combo -> power-ups
    |
Wave manager -> upgrade draft -> next wave / boss
    |
Render arena -> entities -> effects -> boss bar -> HUD

The game uses requestAnimationFrame and delta-time movement. A 0.033-second cap prevents large jumps after a browser tab returns from the background. Data objects define character and enemy balance separately from runtime logic.

5. AI-Assisted Architecture

An AI coding agent inspected the original static repository and helped convert the assignment brief into modules: character selection, Canvas engine, screen states, multi-input controls, wave management, collision detection, documentation, and tests.

A framework-free implementation was selected because the project has one game scene, no server state, and an existing GitHub Pages deployment. This reduced dependencies and kept the source easy to review.

6. AI-Assisted Problem Solving

Collision detection

Circle-based collision checks are used for bullets, enemies, the player, power-ups, and the thesis: distance(a, b) < radius(a) + radius(b).

Input unification

Keyboard and on-screen controls write to the same input state. Mouse and touch use the same firing logic, while keyboard and mobile skill buttons call the same ability and dash functions.

Upgrade and enemy lifecycle

The upgrade overlay pauses simulation without stopping rendering. Enemy IDs and reverse iteration keep collision handling stable when Replicators create new enemies. Boss projectiles use a separate collection because they obey different collision rules.

Browser audio

Web Audio is initialized only after the Begin Defense button is pressed, satisfying browser autoplay rules. Short synthesized tones avoid copyrighted assets and network downloads.

8. Challenges

Character and upgrade balance is stored as data so it can be tuned without rewriting game logic. Mobile input required auto-aim and dedicated skill buttons. Readability is supported through color, shape, labels, shields, health bars, a boss bar, particles, screen shake, and sound.

9. Limitations

  • The game currently contains one arena.
  • Difficulty needs broader user playtesting.
  • Best scores remain local to one browser.

10. Reflection

The AI agent was most useful for translating requirements into a testable architecture and checking edge cases across game states. Human judgment was still necessary for scope, visual style, honest documentation, and deciding whether each feature improved the product.

The result goes beyond an isolated prompt: it has explicit requirements, persistent state, input handling, collision rules, time-based movement, responsive controls, documentation, and a repeatable test plan.
