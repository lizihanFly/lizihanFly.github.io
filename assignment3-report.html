1. Objective

This assignment compares one file-analysis task across a deterministic browser agent, an online OpenAI-compatible model, and a local model served by Ollama. The same interface, file, and question are used in each mode, making privacy, setup, speed, and output easier to compare.

The provider-adapter pattern keeps the interface independent of a specific model. The uploaded file is read with FileReader, limited to 2 MB, and truncated before a network request to prevent oversized prompts.

4. Online Agent

The online mode works with an OpenAI-compatible chat-completions endpoint. The default example uses https://api.deepseek.com/chat/completions and deepseek-chat. The user enters a key, tests the connection, adds a file, and asks a focused question.

Security decision: the API key remains in page memory only. It is never saved to local storage, cookies, source files, or Git.

A production public service should place permanent credentials behind a server-side proxy. Some model providers also block direct browser requests through CORS policy.

5. Local Ollama Deployment

The local mode calls Ollama's /api/chat endpoint. Ollama was not installed on the development computer when checked on June 10, 2026, so this report does not invent a successful local run.

ollama pull qwen2.5:3b

After starting Ollama, select Local Ollama, confirm the endpoint and downloaded model name, then use the connection test. In this mode file content remains on the user's computer.

6. IDE Integration

An AI coding agent inspected the existing GitHub Pages repository, helped translate the rubric into features, proposed the provider-adapter architecture, implemented drag-and-drop input and error states, and reviewed the result. Human decisions preserved the static architecture, prohibited committed secrets, and required honest documentation of unverified external services.

Provider-specific code was separated into queryOnline() and queryOllama(), while buildMessages() creates the shared grounded prompt. This reduced duplication and made the modes easier to test.

7. Browser Baseline

The no-key mode is clearly labeled as a rule-based tool, not an LLM. It provides document statistics, keyword extraction, risk and TODO detection, code structure inspection, CSV profiling, and question-to-sentence ranking. This creates a reliable baseline and keeps the live demonstration usable without credentials.

9. Reflection

FactorOnline modelLocal model
SetupAPI account and keyInstall Ollama and download model
QualityOften stronger reasoningDepends on model and hardware
PrivacyContent leaves computerContent remains local
CostPossible per-token feeNo per-request fee
AvailabilityRequires network/providerWorks offline after download

The online model is easiest for strong analysis on modest hardware. Ollama is preferable for private files and offline work. The browser baseline is most predictable and explainable, but cannot match an LLM's semantic reasoning.

10. Conclusion

Agent deployment is more than choosing a model. Secret handling, adapters, validation, error reporting, privacy, and documentation determine whether an agent is useful in a real workflow.
